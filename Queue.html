Queue


Queues are the opposite of Stacks, but they share a lot of similarities. They are both mutable, and they have similar initializations. The big difference between the queue and the stack is that the queue has a First In First Out (FIFO) iteration instead of the FILO iteration of the stack. A line of cars at a stoplight is a good visualization of a queue. The first car that got there is going to be the first one out once the light turns green. Queues are used in various ways, such as keeping track of orders in fast food restaurants. Their methods are functionally identical to those of the Stack, but some of them have different names, as we will see below.

Initialization

Remember to import java.util

The initialization for the Queue is once again very similar to that of an ArrayList.


LinkedList < Object > myQueue = new LinkedList< Object >();

LinkedList < Object > myQueue = new LinkedList<>();


You cannot initialize a Queue with a set number of starting slots because it is an Abstract Data Type.

You might be wondering why I put LinkedList instead of Queue. This is because the LinkedList is a type of Queue (a child class of the Queue), and it is the most straightforward one for learning purposes. If we tried to initialize it like this


Queue < Object > myQueue = new Queue< Object >();


Then we would receive a compilation error from the computer, since the Queue class is an abstract parent class (abstract classes can't be initialized).

Useful Methods

.add(Object o) - places an object on the end of the queue.

.peek() - returns the head of the queue, but does not remove it.

.poll()-returns and removes the head of the queue.

.remove()-simply returns head of queue.

Iteration

Since the Queue is so similar to the Stack, this means that their iteration is roughly the same.


while(myQueue.peek()!=null)

{

myQueue.peek();

//code

myQueue.pop();

}


The one apparent difference is that instead of myQueue.empty() == false in the while condition, we check if the myQueue.peek() != null. This tells our while loop to keep working until it sees a null reference, which is basically the same thing as checking if the Queue is empty. We should do this instead of using .empty() because the abstract parent class of Queue does not have an empty() method. However, if you happen to be using a child class of Queue that does have empty(), then you can use empty(). This is a visualization of the Queue.
